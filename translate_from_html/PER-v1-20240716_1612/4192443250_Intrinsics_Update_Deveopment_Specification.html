Intrinsics Update Deveopment Specification

Requirements

Because the mass production car currently has the original internal reference of Baolong in Persistent's viewing camera, this affects the stitching diagram of the environment, which will affect the attitude of the parking.The/data/callib/dump or the cloud backup stores our optimized internal reference. The project expects the optimized internal reference to be updated into the Persistent through this OTA.

Plan discussed by the project team

Because the local Folder (/Data/Calib/Dump/XXTRINSIC) or Yaml files in the cloud backup can be obtained in the correct camera.Therefore, this plan is to get the corresponding camera internal reference from the local Folder or the cloud, thereby updating the calibration parameters in the Persistent.

Acceptable risk

Due to time urgency, the software may not be fully tested.But after discussion, the acceptable risks are:

  1. Due to various reasons, CRASH cannot run, which causes the unable to achieve the expected effect

  2. The procedure has no CRASH, but it cannot be successfully updated

  3. There are still some vehicles that have the correct camera internal reference without the local Folder or the backup on the cloud, but the part of the vehicle cannot update the internal reference of the camera

Unacceptable risk

The program can accept the procedure cannot be successfully updated, but it cannot be accepted:

  1. CPU LOAD is too high, so that the system cannot run;

  2. Update failure and causes the original parameters in Persistent to lose

  3. Destruction file system

Overall Design

System Architecture

  1. After the OTA is upgraded, the backup parameters of all vehicles in the cloud will be decompressed under/asw/calib/, named after UUID, and the UUID of each car corresponds to a car.According to the self -car UUID, you can quickly get the cloud parameters corresponding to the car

  2. IntrinsicupDate program is automatically pulled up by ESME

  3. IntrinsicupDate will read the calibration parameter from the Persistent Server, and find the corresponding YAML file from the two directory of the/data/call and/asw/Calib. If you can find the yaml file at any directory, you will pass the relevant verification.Update the internal parameters in YAML to the Persistent Server

  4. In order to ensure that the program is only successfully executed once, it is necessary to write the successfully executed logo in the Persistent Server; when the next start is started, check the bit

Note: /Data/shadow-VEHICLE-Info.json can read UUID, but the premise is that Shadow's service can start

Workflow of Intrinsic Update in OTA2

In the core module of this function, INTRINSICUPDATE, and the PIPELINE after the power is called:

  1. Read UpdateFlag from Persistent Server. If you read Status_default, enter the next step; otherwise, quit this program

  2. Read the SensorxXXTRINSIC structure from Persistent Server.

  3. Call the function RET_VALUE = Updatexxtrinsic (Const SensorXXTRINSIC & OLD_PARAMS, Sensorxxtrinsic & New_params)

    1. Read the/data/Calib/DUMP/XXTRINSIC directory. If the directory exists, jump to C;

    2. According to the self -car UUID, obtain the/asw/Calib/$ UUID directory, if the directory exists, enter the next step; if it does not exist, exit and return to the status_file_not_found;

    3. Analyze the yaml file obtained in A or B. If all the fields are successfully parsed, the next step is to be performed; if the analysis is incorrect, it will exit and return to the status_file_corrupted;

    4. Compare yaml parameters and OLD_PARAMS, according to the comparison results:

      1. If all parameters (including CX/CY) are consistent, then exit and return status_param_same;

      2. If all parameters are the same except to look around CX/Cy, enter the next step;

      3. If there are still other parameters in addition to looking around CX/CY, then exit and return to Status_param_check_failed

    5. Copy the OLD_PARAMS to new_params, and use yaml's CX/CY to update the new_params CX/CY, exit and return the status_success

  4. If RET_VALUE is status_success, Write new_params into persisten and enter the next step

  5. Place UpdateFLAG to Ret_value and write to the Persistent to exit the process.

Workflow of Intrinsic Update in OTA3

In the core module of this function, INTRINSICUPDATE, and the PIPELINE after the power is called:

  1. Read updateFlag from Persistent Server.

  2. Read the SensorxXXTRINSIC structure from Persistent Server.

  3. Call the function RET_VALUE = Updatexxtrinsic (Const SensorXXTRINSIC & OLD_PARAMS, Sensorxxtrinsic & New_params)

    1. According to the self -car UUID, obtain the/asw/Calib/$ UUID directory. If the directory exists, jump to D;

    2. Read the/data/callib/dump/xxtrinsic directory. If the directory exists, jump to D;

    3. Read the/data/asw/callib/dump/xxtrinsic directory. If the directory exists, jump to D, if not exist, then exit and return to the status_file_not_found;

    4. Analyze the yaml file obtained in A or B or C. If all fields are successfully parsed, the next step is to be performed.

    5. Compare yaml parameters and OLD_PARAMS, according to the comparison results:

      1. If all parameters (including CX/CY) are consistent, then exit and return status_param_same;

      2. If all parameters are the same except to look around CX/Cy, enter the next step;

      3. If there are still other parameters in addition to looking around CX/CY, then exit and return to Status_param_check_failed

    6. Copy the OLD_PARAMS to new_params, and use yaml's CX/CY to update the new_params CX/CY, exit and return the status_success

  4. If RET_VALUE is status_success, Write new_params into persisten and enter the next step

  5. Place UpdateFLAG to Ret_value and write to the Persistent to exit the process.
