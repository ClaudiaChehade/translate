runtime analysis:

  • openvins has ~3x higher output frequency than vins mono → aligned them by image time

    • slightly difference

      • openvins ~2.75s leading time

        • it has dt_slam_delay: 2 # delay before initializing (helps with stability from bad initialization...)

        • it has init_window_time: 0.75 # how many seconds to collect initialization information

        • it has calibrate the time delay between imu and camera, and t_cam = t_imu - t_calibrate

      • vins mono use the image time from ros message directly

  • time selection

    • openvins → whole execution time of function VioManager :: track_image_and_update

    • vins mono → whole execution time of each measurement in the "for loop" of estimator_node.cpp :: process. only vins_estimator node.
